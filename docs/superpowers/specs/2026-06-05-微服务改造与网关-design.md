# 微服务改造与网关 设计文档

- **日期**:2026-06-05
- **状态**:已确认,待编写实现计划
- **关系**:本文档是对《2026-06-05-用户权限登录系统-design.md》的**架构演进**。原设计的功能需求(RBAC、四种登录、Swagger 等)不变,本文档只改变**部署/运行架构**:单体 → 微服务。

## 1. 目标与驱动

把现有单体的用户权限登录系统改造为微服务架构,新增 API 网关。

**驱动(场景 B)**:auth 是更大平台的第一个服务,后续会加订单、商品等业务服务,需要**统一网关入口**和**统一鉴权**。本次改造为后续服务铺路:新增业务服务只需注册到 Nacos + 在网关加一条路由,即可复用统一鉴权,无需重复写登录校验。

**非目标(YAGNI)**
- 不做服务拆分到极致(不为 auth 内部再拆"用户服务/权限服务");auth 业务暂留在一个 `auth-service`。
- 不引入链路追踪、熔断限流、消息队列等(后续按需再加)。
- 本期只用一个 Nacos namespace(`test`,即共享 Nacos 上的 `dy_wjb` 命名空间),不做多环境矩阵。

## 2. 技术栈

| 维度 | 选型 |
|---|---|
| 版本基线 | Spring Boot 3.2.x + Spring Cloud 2023.x + Spring Cloud Alibaba 2023.x(配套锁定) |
| 注册/配置中心 | Nacos(`192.168.0.230:8848` HTTP / `9848` gRPC,账号 `nacos/nacos`) |
| 网关 | Spring Cloud Gateway(响应式 WebFlux) |
| 鉴权 | Sa-Token:网关用 `sa-token-reactor-spring-boot3-starter`,服务用 `sa-token-spring-boot3-starter`;共享 Redis |
| 服务调用 | OpenFeign(本期暂无跨服务调用,先具备能力) |
| 持久化 | MySQL(auth-service)+ Redis(共享会话) |
| 构建 | Maven 多模块(父 pom 统一版本管理) |

## 3. 模块拆分与架构

### 3.1 目录结构(Maven 多模块)

```
auth-wjb/                      # 父工程(packaging=pom)
├── pom.xml                    # 父 pom:dependencyManagement 锁定 Boot/Cloud/Alibaba/Sa-Token 版本
├── auth-common/              # 公共模块(jar)
├── auth-gateway/             # 网关服务(可执行 jar)
├── auth-service/             # 认证业务服务(可执行 jar)
├── frontend/                 # 前端(后续里程碑)
├── sql/                      # 建表脚本
└── docs/                     # 文档
```

> 原单体 `backend/` 目录将被这三个模块取代:`backend/` 的代码按职责拆入 `auth-common` 与 `auth-service`。

### 3.2 模块职责

| 模块 | 类型 | 职责 | 依赖中间件 |
|---|---|---|---|
| `auth-common` | jar 库 | 复用代码:`Result`、`ResultCode`、`BaseEntity`、异常、错误码、通用常量(如请求头名 `X-User-Id`) | 无 |
| `auth-gateway` | 可执行 | 唯一对外入口;路由转发(`lb://`);统一校验登录态 + 权限;通过后注入 `X-User-Id` 转发;注册 Nacos | Nacos、Redis |
| `auth-service` | 可执行 | 登录(写 Sa-Token 会话 + 权限快照到共享 Redis)、用户/角色/菜单 CRUD;信任网关不重复校验登录;注册 Nacos | Nacos、Redis、MySQL |

### 3.3 请求链路

```
前端 → auth-gateway(8080,统一鉴权)→ auth-service(8081,业务)
                  ‖ 共享 Redis(Sa-Token 会话 + 权限快照)
             Nacos(注册发现 + 配置中心)
```

- 对外端口收归网关 **8080**;auth-service 改 **8081**,仅经网关访问(生产不对外暴露)。

## 4. 网关统一鉴权

### 4.1 Sa-Token 微服务模式
- **共享 Redis**:gateway 与 auth-service 连同一 Redis,会话数据互通。
- gateway 用响应式集成 `sa-token-reactor-spring-boot3-starter`;auth-service 用 `sa-token-spring-boot3-starter`。

### 4.2 登录(写会话 + 权限快照)
```
前端 → 网关 → auth-service: POST /auth/login
  校验账密通过
  → StpUtil.login(userId)                                  # 会话写共享 Redis
  → 查用户权限码集合, StpUtil.getSession().set("perms", permList)   # 权限快照存 Redis
  → 返回 token
```

### 4.3 请求鉴权(网关 SaReactorFilter)
```
前端带 token → 网关过滤器:
  1. 放行白名单:/auth/login、/auth/sms-code、/auth/email-code、/auth/oauth/**、文档、健康检查
  2. StpUtil.checkLogin()                                  # 读共享 Redis 校验登录态
  3. 路由→权限码规则:
       SaRouter.match("/system/user/**").check(r -> StpUtil.checkPermission("system:user:*"))
       # 权限码从会话快照读,网关不连 MySQL
  4. 通过 → 注入请求头 X-User-Id → 转发 auth-service
  5. 失败 → 网关直接返回统一 401/403 JSON,不进下游
```

### 4.4 下游服务(auth-service)
- 移除 P1 的 `SaTokenConfig`(MVC 登录拦截器);鉴权已上移网关。
- 业务从请求头 `X-User-Id` 取当前用户,封装 `UserContext` 工具。
- 审计字段(`create_by`/`update_by`)的"当前用户"改为从 `X-User-Id` 读,不再依赖 `StpUtil`。

### 4.5 关键权衡(选 A:全在网关)
1. **权限为登录时快照**:权限码在登录瞬间算好存 Redis。改某用户权限需其重新登录(或提供"刷新会话权限"接口)才生效。换取网关响应式、不阻塞、不连库。
2. **网关维护"路由→权限码"映射**:新增业务服务时,其权限规则需在网关登记。规则集中在网关一个配置类,统一维护。

## 5. Nacos 注册与配置

### 5.1 服务注册发现
- gateway、auth-service 启动注册到 Nacos。
- 网关路由按服务名负载均衡:
  ```yaml
  routes:
    - id: auth-service
      uri: lb://auth-service
      predicates:
        - Path=/auth/**,/system/**
  ```
- 新增业务服务:注册 Nacos + 网关加一条路由即接入。

### 5.2 配置中心(配置外置)
- 敏感/环境配置上移 Nacos,本地仅留最小 `bootstrap` 配置(服务名 + Nacos 地址 + namespace)。
- git 仓库零数据库密码。

| Data ID | 内容 | 使用方 |
|---|---|---|
| `auth-common.yml` | Redis、Sa-Token、日志(共享) | 所有服务 |
| `auth-service.yml` | MySQL、MyBatis-Plus、auth 专有 | auth-service |
| `auth-gateway.yml` | 路由规则、鉴权白名单 | auth-gateway |

### 5.3 环境隔离
- 本期 namespace 用 **`test`**(共享 Nacos 的 `dy_wjb` 命名空间,id=`test`);`group` 用 `DEFAULT_GROUP`。
- 后续可加独立 `dev`/`prod` namespace。

## 6. 迁移路径(渐进,每步可验证)

现状:代码仅到 P1(骨架 + 健康检查 + 公共基础 + 建表脚本),P2(登录/用户管理)尚未实现。故本次为 P1 基础上的重构,P2 业务直接落 `auth-service` 编写。

```
1. 建父 pom(packaging=pom),抽公共版本管理
2. 建 auth-common:搬入 P1 的 Result/ResultCode/BaseEntity/异常,做成 jar
3. 建 auth-service:
   - 搬入 AuthApplication、MyBatis-Plus 配置、MetaObjectHandler、HealthController
   - 移除 SaTokenConfig(MVC 拦截器)
   - 接入 Nacos discovery + config(bootstrap)
   - 审计字段当前用户改为从 X-User-Id 读
4. 建 auth-gateway:
   - Spring Cloud Gateway + sa-token-reactor + Nacos
   - 路由(lb://auth-service)+ SaReactorFilter 统一鉴权 + 白名单
5. Nacos 建 3 个配置 Data ID
6. 联调:Nacos → auth-service → gateway,经网关访问 /health 与受保护接口
```

## 7. 验证标准

- 两服务在 Nacos 控制台可见(已注册)。
- `GET http://网关:8080/health` 经路由到 auth-service 返回 `{"code":200,"data":"UP"}`。
- 未登录访问受保护接口 → **网关**直接返回 401(请求未进 auth-service)。
- 登录后带 token → 网关校验通过 → 下游正常响应。
- auth-service 直连端口(8081)仅内网,不对外。

## 8. 测试策略

- **auth-common**:纯单元测试(Result/异常),无中间件依赖。
- **auth-service**:`@SpringBootTest` + MockMvc;通过 `spring.cloud.nacos.config.import-check.enabled=false` + 测试本地配置,使测试不强依赖在线 Nacos。
- **auth-gateway**:响应式过滤器以集成联调为主,脚本化 curl 验证三场景(白名单放行 / 未登录拦截 / 登录后放行)。

## 9. 风险点

1. **版本兼容**:Boot 3.2.x ↔ Spring Cloud 2023.x ↔ Spring Cloud Alibaba 2023.x ↔ Sa-Token reactor 版必须配套,实现时锁定一组验证过的版本号。
2. **网关 WebFlux ≠ 服务 WebMVC**:网关内禁用阻塞式 JDBC,权限走 Redis 快照(见 4.5)。
3. **gateway 也需连 Redis**(读 Sa-Token 会话)。

## 10. 里程碑拆分(交给 writing-plans)

- **M1(本设计核心)**:多模块骨架 + Nacos 注册配置 + 网关统一鉴权,把 P1 跑通到"经网关访问 /health + 未登录拦截"。
- **M2**:在 auth-service 落地 P2 业务(账密登录 + 用户管理),打通"登录→网关鉴权→下游"。
- 后续:验证码登录、OAuth、前端(沿用原路线 P3/P4/P5,落到 auth-service / 网关白名单)。
