# 用户权限登录系统 设计文档

- **日期**:2026-06-05
- **状态**:已确认,待编写实现计划
- **作者**:与 Claude 协作(brainstorming)

## 1. 目标与范围

构建一个**通用后台管理系统的权限底座**:前后端一体,实现用户认证(登录)与基于角色的访问控制(RBAC 授权)。可作为后续业务系统长期复用、维护的基础。

**核心需求清单**

- 权限模型:基础 RBAC(用户 → 角色 → 权限/菜单)
- 认证方式:JWT(token 模式,持久化到 Redis)
- 登录方式(四种,统一收口):
  1. 账号密码登录(带图形验证码)
  2. 手机/邮箱验证码登录
  3. 第三方 OAuth 登录(微信 / GitHub)
  4. 图形验证码(防爆破)
- 接口文档:Swagger(springdoc-openapi + Knife4j)
- 前端:后台管理界面(动态菜单 + 按钮级权限)

**非目标(YAGNI,本期不做)**

- 数据权限(只控制能看哪些数据)——预留 `dept_id` 字段,本期不实现
- 多租户 / ABAC 属性权限
- 工作流、审批等业务功能

## 2. 技术栈

| 层 | 技术 |
|---|---|
| 前端 | Vue 3 + Vite + Element Plus + Pinia + Vue Router + Axios |
| 后端 | Spring Boot 3.x + Sa-Token + MyBatis-Plus |
| 存储 | MySQL 8(业务数据) + Redis(登录态 / 验证码 / OAuth state) |
| 鉴权 | Sa-Token(token 模式,持久化到 Redis) |
| 接口文档 | springdoc-openapi + Knife4j(访问 `/doc.html`) |
| 测试 | JUnit 5 + Mockito + MockMvc;集成测试用 H2 / Testcontainers |

**选型理由**:Sa-Token 相比 Spring Security,RBAC、踢人下线、多端登录、注解鉴权、Redis 集成开箱即用,API 简洁(`StpUtil.login(id)` 一行登录),中文文档,对国内后台场景贴合,开发效率高且代码可读性好。

## 3. 整体架构

### 3.0 仓库目录结构(单仓库前后端分目录)

```
auth-wjb/
├── backend/          # Spring Boot 后端(端口 8080)
│   ├── src/main/java/...
│   ├── src/main/resources/
│   └── pom.xml
├── frontend/         # Vue 3 前端(Vite 开发端口 5173,代理转发后端)
│   ├── src/
│   ├── package.json
│   └── vite.config.ts
├── docs/             # 设计文档
├── sql/              # 建表脚本
└── README.md
```

前后端各自独立构建、独立启动;开发期前端用 Vite 代理把 `/api` 转发到后端 8080。

### 3.1 后端模块划分(单体分包,职责单一)

```
backend/src/main/java/.../
├── config/         # Sa-Token 配置、跨域、全局异常、MyBatis-Plus 配置、Knife4j 配置
├── controller/     # 接口层:AuthController / UserController / RoleController / MenuController
├── service/        # 业务层:认证、用户、角色、菜单、验证码、OAuth
├── mapper/         # MyBatis-Plus Mapper
├── entity/         # 数据库实体(含 BaseEntity 基类)
├── dto/            # 请求/响应对象(登录请求、分页查询等)
├── enums/          # 枚举(状态、登录类型、错误码)
├── common/         # 统一返回 Result、分页、工具类
└── security/       # Sa-Token 鉴权拦截、登录态工具封装
```

### 3.2 设计原则

- **认证与授权分离**:登录逻辑集中在 `AuthService`;权限校验交给 Sa-Token 注解(`@SaCheckPermission`)。
- **多种登录方式统一出口**:账密 / 验证码 / OAuth 三类登录,前置校验逻辑各异,但都收口到同一句"校验通过 → `StpUtil.login(userId)` → 返回 token"。新增登录方式不改主流程。
- **权限数据驱动**:菜单/按钮权限存数据库,登录时把用户的权限码集合加载进 Sa-Token Session;前端按权限码动态渲染菜单与按钮。

## 4. 数据库设计

### 4.1 公共字段(BaseEntity,所有表统一继承)

| 字段 | 类型 | 说明 |
|---|---|---|
| create_time | datetime | 创建时间,插入时 MyBatis-Plus 自动填充 |
| update_time | datetime | 更新时间,更新时自动填充 |
| create_by | bigint | 创建人 id,从 Sa-Token 当前登录用户自动取 |
| update_by | bigint | 更新人 id,自动取 |
| deleted | tinyint | 逻辑删除:0 未删 / 1 已删,默认 0,`@TableLogic` |

落地:抽 `BaseEntity` 基类;`create_time/update_time/create_by/update_by` 用 `MetaObjectHandler` 自动填充;`deleted` 用 `@TableLogic`,查询自动带 `deleted=0`。

`role_id` + `menu_id`(联合唯一) + 公共字段

| 字段 | 类型 | 说明 |
|---|---|---|
| id | bigint PK | |
| user_id | bigint | 关联本系统用户 |

### 4.8 权限关系流

```
用户 → (sys_user_role) → 角色 → (sys_role_menu) → 菜单/权限码
```

登录时一次性查出用户所有 `perm` 权限码集合,塞进 Sa-Token Session,鉴权时比对。

> 短信/邮箱验证码、图形验证码、OAuth state **不进数据库**,全放 Redis 带过期时间(验证码 5 分钟、图形验证码 2 分钟)。

## 5. 接口设计

### 5.1 认证接口(`/auth/**`,免登录访问)

| 方法 | 路径 | 说明 |
|---|---|---|
| GET | `/auth/captcha` | 获取图形验证码(返回 base64 图 + captchaKey) |
| POST | `/auth/sms-code` | 发送手机验证码(需先过图形验证码) |
| POST | `/auth/email-code` | 发送邮箱验证码 |
| POST | `/auth/login` | 账号密码登录(带图形验证码校验) |
| POST | `/auth/login/sms` | 手机验证码登录 |
| GET | `/auth/oauth/{provider}/url` | 获取第三方授权跳转地址(生成 state 存 Redis) |
| GET | `/auth/oauth/{provider}/callback` | 第三方回调,校验 state → 登录或引导绑定 |
| POST | `/auth/logout` | 退出登录(`StpUtil.logout()`) |
| GET | `/auth/userinfo` | 获取当前登录用户信息 + 权限码 + 菜单树 |

### 5.2 业务管理接口(需登录 + 权限码)

```
/system/user/**    用户管理   @SaCheckPermission("system:user:xxx")
/system/role/**    角色管理   @SaCheckPermission("system:role:xxx")
/system/menu/**    菜单管理   @SaCheckPermission("system:menu:xxx")
```

每类含 `list / {id} / add / update / delete` 标准 CRUD,分别对应 `query/add/edit/remove` 权限码。

### 5.3 统一返回结构

```json
{ "code": 200, "msg": "成功", "data": {} }
```

所有接口走全局 `Result<T>` 包装;异常由全局 `@RestControllerAdvice` 兜底转成统一结构。

### 5.4 接口文档

- springdoc-openapi + Knife4j,访问 `/doc.html`。
- 控制器/DTO 上用 `@Tag`、`@Operation`、`@Schema` 写中文说明。
- Sa-Token 拦截器需**放行**:`/doc.html`、`/v3/api-docs/**`、`/swagger-ui/**`、`/webjars/**`。

## 6. 核心鉴权流程

### 6.1 账号密码登录

```
前端提交 username + password + captchaKey + captchaCode
  → 校验图形验证码(Redis 取 captchaKey,比对后删除)
  → 查 sys_user,BCrypt 校验密码
  → 校验 status 是否禁用
  → StpUtil.login(userId)         // Sa-Token 生成 token,写 Redis
  → 查该用户所有权限码,存入 StpUtil.getSession()
  → 返回 token
```

### 6.2 短信验证码登录

```
发码:校验图形验证码 → 生成 6 位码存 Redis(5 分钟) → 调短信网关
登录:取 Redis 码比对 → 用户不存在则自动注册 → StpUtil.login(userId) → 返回 token
```

(邮箱验证码登录同理,走邮件网关。)

### 6.3 第三方 OAuth 登录(以 GitHub 为例)

```
获取地址:生成随机 state 存 Redis(5 分钟) → 拼授权 URL 返回前端
回调:校验 state → 用 code 换 access_token → 拉第三方用户信息(openId)
  → 查 sys_oauth_binding:
       已绑定 → StpUtil.login(关联 userId)
       未绑定 → 返回"需绑定"标识,引导用户登录后绑定,或自动创建新用户
```

### 6.4 请求鉴权(登录后每个请求)

```
Sa-Token 拦截器校验 token 有效性(Redis)
  → @SaCheckPermission("system:user:add") 注解校验权限码
  → 无权限抛 NotPermissionException,全局异常转 403 统一返回
```

### 6.5 前端权限控制

- 路由守卫:无 token 跳登录页。
- 登录后拿 `userinfo` 的菜单树 → 动态生成路由和侧边栏。
- 按钮级:`v-permission="'system:user:add'"` 自定义指令,无权限码则隐藏按钮。

## 7. 安全设计

- **密码**:BCrypt 加盐哈希,数据库永不存明文。
- **防爆破**:登录失败计数(Redis),同一账号连续失败 5 次锁定 10 分钟;图形验证码挡机器批量。
- **验证码安全**:一次性,校验后立即从 Redis 删除;短信码限频(同手机号 60 秒只能发一次)。
- **Token**:Sa-Token token 存 Redis 设过期(如 7 天),支持续签;敏感操作可二次校验。
- **OAuth state**:防 CSRF,回调必校验 state。
- **越权防护**:所有业务接口强制 `@SaCheckLogin` + `@SaCheckPermission`,杜绝接口裸奔。
- **跨域**:统一 CORS 配置,生产限定前端域名。

## 8. 错误处理(全局统一)

| 异常 | HTTP/code | 返回 |
|---|---|---|
| `NotLoginException`(未登录/token 失效) | 401 | "登录已过期,请重新登录" |
| `NotPermissionException`(无权限) | 403 | "无操作权限" |
| `NotRoleException`(无角色) | 403 | "无访问权限" |
| 业务异常 `ServiceException`(自定义) | 200/500 | 携带业务提示语 |
| 参数校验异常(`@Valid`) | 400 | 字段级错误提示 |
| 兜底 `Exception` | 500 | "系统繁忙",同时记日志 |

全部由 `@RestControllerAdvice` 全局异常处理器收口成统一 `Result`。
