# M6:验证码登录与防爆破 设计文档

- **日期**:2026-06-05
- **状态**:已确认,待编写实现计划
- **关系**:在已完成的 RBAC 系统(P1–M5)上扩展登录方式,属《用户权限登录系统》原 spec 的"登录方式"落地。

## 1. 目标与范围

为现有系统增加:图形验证码、手机/邮箱验证码登录、账密登录防爆破。本里程碑**前后端全包**。

**关键决策(已与用户确认)**
- 短信/邮箱**全部 mock**:验证码生成、存储、限频、校验逻辑全真,仅"发送"步骤打日志 + 开发环境接口回传 code;发送器留可插拔接口,日后接真实网关只换实现。
- 图形验证码用 **easy-captcha**;**账密登录**与**发送短信/邮箱码**前都要校验图形码。
- 手机/邮箱验证码登录:账号**必须已存在**(手机号/邮箱未注册则拒绝,不自动创建)。
- 手机和邮箱**都支持**验证码登录(对称)。
- 防爆破:账密**连续失败 5 次 → 锁定 10 分钟**。

**非目标(YAGNI)**
- 不接真实短信/邮件网关(留接口)。
- 不做滑块/行为验证码(图形算术码足够)。
- 不改 OAuth(M7)。

## 2. 技术栈与依赖

- 后端:沿用 auth-service(Spring Boot + Sa-Token + MyBatis-Plus + Redis);新增 `com.github.whvcse:easy-captcha`。
- 前端:沿用 Vue3 + Element Plus。
- 存储:验证码与防爆破计数全部走 **Redis**(已有)。

## 3. 接口设计(auth-service,网关白名单放行)

| 方法 路径 | 说明 |
|---|---|
| GET `/auth/captcha` | 生成图形验证码,返回 `{ captchaKey, imageBase64 }`,答案存 `captcha:{key}`(2 分钟) |
| POST `/auth/sms-code` | 发手机码:校验图形码 → 校验手机号已注册 → 限频 60s → 生成 6 位码存 `sms:{phone}`(5 分钟)→ mock 发送 |
| POST `/auth/email-code` | 发邮箱码:同上,针对邮箱,存 `email:{email}` |
| POST `/auth/login` | **改造**:账密登录增加图形验证码校验 + 防爆破锁定 |
| POST `/auth/login/sms` | 手机验证码登录:校验 `sms:{phone}` → 按手机号查用户(必须存在)→ 登录发 token |
| POST `/auth/login/email` | 邮箱验证码登录:校验 `email:{email}` → 按邮箱查用户 → 登录发 token |

**网关白名单**:`/auth/login`、`/auth/login/**`、`/auth/sms-code`、`/auth/email-code` 已放行;**新增放行 `/auth/captcha`**。

**Redis key 约定**

| key | TTL | 用途 |
|---|---|---|
| `captcha:{key}` | 2 分钟 | 图形验证码答案 |
| `sms:{phone}` / `email:{email}` | 5 分钟 | 短信/邮箱验证码 |
| `sms:limit:{phone}` / `email:limit:{email}` | 60 秒 | 发送限频 |
| `login:fail:{username}` | 10 分钟 | 账密失败计数 |
| `login:lock:{username}` | 10 分钟 | 账号锁定标记 |

## 4. 核心流程

### 4.1 图形验证码
- easy-captcha 生成算术验证码(如 "3+5=?",答案 "8"),输出 base64 图。
- `captchaKey` 用 UUID;答案存 Redis(2 分钟)。
- 校验:取 `captcha:{key}` 比对(忽略大小写),**用完即删**;错误或过期均报错。

### 4.2 发送验证码(短信/邮箱对称)
```
POST /auth/sms-code { phone, captchaKey, captchaCode }
  1. 校验图形码(取 captcha:{key} 比对后删除)
  2. 校验手机号已注册(查 sys_user.phone;不存在 → "该手机号未注册")
  3. 限频:存在 sms:limit:{phone} → "发送过于频繁";否则设 sms:limit:{phone} TTL 60s
  4. 生成 6 位数字码 → 存 sms:{phone} TTL 5 分钟
  5. mock 发送:SmsSender.send(phone, code)(默认打日志);dev 环境接口 data 回传 code
```
邮箱流程同理(`email:{email}`、`MailSender`)。

### 4.3 验证码登录
```
POST /auth/login/sms { phone, code }
  1. 取 sms:{phone} 比对;错/过期 → 报错;成功后删除(一次性)
  2. 按 phone 查 sys_user(必须存在,否则"账号不存在")+ 校验未禁用
  3. doLogin(user):StpUtil.login(userId) + 权限快照入会话 → 返回 token
```
邮箱登录按 email 查,流程相同。

### 4.4 账密登录改造 + 防爆破
```
POST /auth/login { username, password, captchaKey, captchaCode }
  1. 检查锁定:存在 login:lock:{username} → "账号已锁定,请 10 分钟后再试"
  2. 校验图形验证码
  3. 校验账密:
     失败 → login:fail:{username} +1(TTL 10 分钟);计数达 5 → 设 login:lock:{username} TTL 10 分钟;抛"用户名或密码错误"
     成功 → 删除 login:fail / login:lock;doLogin(user) → token
```

### 4.5 登录收口
三种登录方式(账密/手机/邮箱)校验通过后统一走 `AuthService.doLogin(user)`:`StpUtil.login(userId)` + 查权限码写入 Sa-Token 会话快照 + 返回 token。新增登录方式不重复登录逻辑。

## 5. 可插拔发送器

```
interface SmsSender { void send(String phone, String code); }
interface MailSender { void send(String email, String code); }
```
默认实现 `LogSmsSender` / `LogMailSender`:打印日志(开发用)。日后接阿里云短信 / JavaMail SMTP 只新增实现类 + 配置切换,业务代码不动。

## 6. 前端改造(登录页)

登录页 `Login.vue` 改造为多方式登录:
- **Tab 切换**:账号密码 / 手机验证码 / 邮箱验证码。
- **账密 Tab**:用户名 + 密码 + **图形验证码**(图片可点击刷新 + 输入框)。
- **手机/邮箱 Tab**:手机号(或邮箱)+ 图形验证码 + 验证码(带"发送"按钮和 60s 倒计时)。
- 调用对应接口(`/auth/captcha`、`/auth/sms-code`、`/auth/login` 等),登录成功后流程与现有一致(存 token → 拉用户数据 → 跳首页)。
- 新增前端 API:`captchaApi`、`smsCodeApi`、`emailCodeApi`、`smsLoginApi`、`emailLoginApi`。

## 7. 错误处理

沿用统一 `Result` + 全局异常。验证码错误、限频、锁定、账号不存在等均通过 `ServiceException` 返回明确中文提示(`code:500`)。

## 8. 测试策略

- **后端离线单测**:验证码校验逻辑、防爆破计数/锁定逻辑(用 Mockito mock 掉 Redis/Mapper,或用嵌入式判断)。
- **后端联调**:启动服务,脚本走 `/auth/captcha`(拿 key+答案,dev 回传)→ 发码(dev 回传 code)→ 验证码登录;以及账密连续错 5 次触发锁定。
- **前端浏览器验证**:登录页三 Tab 切换;图形验证码显示/刷新;手机/邮箱发码倒计时;验证码登录成功进首页。

## 9. 风险与说明

- 全 mock:`dev` 环境接口回传验证码 code 仅为联调便利,**生产环境必须关闭回传**(用 profile 控制,`prod` 不返回)。
- 手机/邮箱登录要求账号已存在,故联调前需保证有带 `phone`/`email` 的测试用户(可在 M6 种子或通过用户管理页补 phone/email)。
