# M9:部署与前端托管 设计文档

- **日期**:2026-06-18
- **状态**:已确认,待编写实现计划
- **关系**:在已完成系统(P1–M8)上做部署形态收口:网关托管前端、`/api` 统一、生产 Nacos 参数化、部署文档。

## 1. 目标与范围

把系统收口成**可部署形态**:产出"内含前端的网关 jar + auth-service jar"+ 部署文档,本机起两个 jar 即可访问整站(单端口)。

**关键决策(已与用户确认)**
- 部署形态:产出可部署产物 + 文档(无目标服务器/Docker,不真部署到远端),本机可验证。
- 前端托管:**网关托管前端静态包**(一端口出整站)。
- 路径统一:后端全部走 **`/api` 前缀**,前端 history 路由占非 `/api` 路径,避免碰撞。
- 生产 Nacos 隔离:**参数化**(namespace/profile 可由启动参数覆盖)+ 文档,不真建空壳 prod 命名空间。

**非目标(YAGNI)**
- 不真部署到远程服务器、不做容器化(无 Docker)。
- 不真建 prod 命名空间(无真实 prod 中间件)。
- 不引入 Nginx(配置作为文档附录供需要时用)。

## 2. 架构变更:`/api` 统一 + 网关托管静态

### 2.1 路径碰撞与解法
前端 history 路由含 `/system/user`、`/dashboard` 等,与后端 `/system/**` 碰撞。解法:**后端统一 `/api` 前缀**,前端路由(非 `/api`)留给 SPA。

### 2.2 网关改造
- **路由**:新增 `/api/**` → `StripPrefix=1` → `lb://auth-service`,取代原直连 `/auth,/system,/health` 路由(写在 Nacos `auth-gateway-dev.yml`)。
- **静态托管**:`dist` 拷入 `auth-gateway/src/main/resources/static/`,WebFlux 静态资源处理器服务之。
- **SPA 回退**:未命中静态文件、非 `/api` 的 GET 请求 → 返回 `index.html`(用 WebFlux `RouterFunction`,order 低于路由/静态);保证 history 刷新不 404。
- **鉴权适配**:`SaReactorFilter` 只拦 `/api/**`;判定时把 `/api` 前缀剥掉得到逻辑路径(如 `/api/auth/login`→`/auth/login`),再走**现有**白名单与动态RBAC 映射(`ApiPermCache.requiredPerm(method, 逻辑路径)`)。**不改 Nacos 映射、不改 auth-service**。静态资源不在 `/api/**` 内,天然不被鉴权拦。

### 2.3 前端改造
- 保持 history 路由;axios `baseURL=/api`(已是)。
- **Vite 代理**改为 `/api → http://localhost:8080` **去掉 rewrite**(让 `/api` 原样到网关,由网关 StripPrefix)。dev/prod 都走 `/api`,一致。
- 构建产物拷到网关 static(见 §4)。

### 2.4 OAuth 生产回调
生产环境前端在网关端口,GitHub 回调地址变为 `http://<网关host>/oauth/callback`,需在 GitHub OAuth App **追加**该生产回调 URL(dev 的 `http://localhost:5173/oauth/callback` 保留)。SPA 回退保证该路径进前端路由。文档说明。

## 3. 生产 Nacos 参数化

bootstrap(auth-service、auth-gateway)的 namespace 与 profile 支持启动参数覆盖,默认不变:
```yaml
spring:
  profiles:
    active: ${SPRING_PROFILES_ACTIVE:dev}
  cloud:
    nacos:
      discovery:
        namespace: ${NACOS_NAMESPACE:test}
      config:
        namespace: ${NACOS_NAMESPACE:test}
```
- 默认 `test/dev`(本机行为不变)。
- 生产:`-DNACOS_NAMESPACE=prod -DSPRING_PROFILES_ACTIVE=prod` → 切 prod 命名空间 + `auth-service-prod.yml`。
- 部署文档写清生产建 `prod` 命名空间 + 配置(模板,密码占位)。

## 4. 构建产物与脚本

- `frontend` 构建 → `dist` → 拷到 `auth-gateway/src/main/resources/static/`(随网关 jar 打包)。
- 一键脚本 `build-all.ps1`(仓库根):`npm --prefix frontend run build` → 清空并复制 `dist/*` 到 `auth-gateway/src/main/resources/static/` → `mvn -f pom.xml -DskipTests package`。产出含前端的网关 jar + auth-service jar。
- `auth-gateway/src/main/resources/static/` 内容加入 `.gitignore`(构建产物不入库),保留 `.gitkeep` 占位。

## 5. 部署文档 `docs/DEPLOY.md`

1. 前置:JDK21;Nacos/MySQL/Redis 可达;需要的配置 Data ID 清单。
2. 生产 Nacos:建 `prod` 命名空间 + `auth-common.yml`/`auth-service-prod.yml`/`auth-gateway-prod.yml`(模板内容,密码占位)。
3. 构建:`build-all.ps1` 或手动步骤(npm build → 拷 dist → mvn package)。
4. 启动:
   - `java -DNACOS_NAMESPACE=prod -DSPRING_PROFILES_ACTIVE=prod -jar auth-service.jar`
   - `java -DNACOS_NAMESPACE=prod -DSPRING_PROFILES_ACTIVE=prod -jar auth-gateway.jar`
   - 网关端口(默认 8080)即整站入口。
5. GitHub OAuth 生产回调 URL 追加说明。
6. 附录:Nginx 反代配置(可选,供前后端分离部署时用)。

## 6. 本机验证

- 跑 `build-all.ps1` → 起 auth-service + auth-gateway(默认 test/dev)→ 浏览器开**网关 8080**:
  - 直接出登录页(静态托管生效)。
  - admin 登录 → 动态菜单 → 用户/角色/菜单管理正常(`/api` 经 StripPrefix 到后端,鉴权剥前缀后正常)。
  - 刷新 `/system/user`(history)→ SPA 回退不 404。
  - playwright 浏览器实测。
- 仅起两个 jar,不依赖 Vite(生产形态)。

## 7. 风险

- 响应式网关静态 + SPA 回退需正确处理 handler order;重点验证"刷新非根路由不 404""静态不被鉴权拦""`/api` 路由优先于 SPA 回退"。
- StripPrefix 与 SaReactorFilter 顺序:鉴权在过滤器链中基于原始 `/api/...` 路径判定(代码内剥前缀),StripPrefix 仅影响转发到下游的路径,二者不冲突。
- 端口冲突:本机验证时确保 8080(网关)、8081(auth-service)空闲。
